Кому нужна доступность в вебе

Визуальные
Пользователи с различными уровнями нарушения зрения, дальтонизмом или полной слепотой. Некоторым требуется программное обеспечение, такое как программы чтения с экрана, чтобы объявить содержимое экрана либо синтезированным голосом, либо шрифтом Брайля.

Базовые принципы

Аудиальные
Пользователи с различными уровнями нарушения слуха. Если приложение использует звук для передачи информации, то также должны быть предоставлены визуальные альтернативы или субтитры.

Introduction to Web Accessibility and W3C Standards

Моторные
Пользователям может быть трудно использовать мышь. Вместо неё они могут использовать клавиатуру или вспомогательные технологии для управления своим устройством.
Когнитивные
Пользователей могут испытывать затруднения с пониманием содержания, информации, пониманием языка, сохранением фокуса или восприятием. Веб-сайты должны быть построены таким образом, чтобы вся информация была легко понятна и была легко доступна.

WCAG
The Web Content Accessibility Guidelines

WCAG 📖
Рекомендации по обеспечению доступности веб-контента

WCAG 1.0: Основное

  • Не полагайтесь на цвет
  • Используйте правильную разметку и стили
  • Разрабатывайте для устройств с разными настройками
  • Обеспечивайте ясную навигацию
  • Делайте контент понятным

Web Content Accessibility Guidelines 1.0

WCAG 2.0: Что изменилось

  • Появилось 4 основых принципа
  • В каждом принципе есть гайдлайны
  • У всех гайдлайнов есть критерии успеха на уровне A, AA, AAA
  • Более точен при ручном и автоматизированном тестировании
  • Применяется в большом количестве веб-технологий и предназначен для применения в будущих технологиях

How WCAG 2.0 Differs from WCAG 1.0

WCAG 2.1: Не заменяет, но дополняет

  • Смена ориентации устройства: сайт доступен в обеих ориентациях, функциональность остаётся на том же уровне
  • Перекомпоновка: при увеличении масштаба сайт перестраивается
  • Дополнительный контент по наведению и фокусу: содержимое должно быть доступно для пользователей, не использующих мышь
  • Метки в имени: доступные имена интерактивных элементов
  • Сообщение о статусе: все изменения озвучиваются

What’s New in WCAG 2.1

WCAG 2.2: Видимый фокус (Черновик)

  • Минимальная область: область индикации фокуса больше или равна самой длинной стороне ограничительного прямоугольника сфокусированного контрола, умноженной на 2px
  • Контрастность фокуса: изменение цвета в фокусе имеет контраст 3:1 с цветом до состояния фокуса
  • Контрастность или толщина: показатель фокуса имеет контрастность 3:1 или его толщина минимум 2px

The Difference Between WCAG A and WCAG AA Conformance? - Level Access

Уровень A

Начальный. Если рекомендации не выполнены, вспомогательные технологии не смогут полностью воспринимать содержимое страницы.

25 критериев успеха

Уровень AA

Универсальная поддержка. Требуется для веб-сайтов правительства и государственных органов ЕС, а также лежит в основе закона ADA Compliance.

45 критериев успеха: Все критерии успеха уровня A + 20 критериев успеха уровня AA

Уровень AAA

Поддержка высокого уровня. Обычно используется сайтами, которые обслуживают специализированную аудиторию.

73 критерия успеха: Все критерии успеха уровня AA + 28 критериев успеха уровня AAA

SOLID
это аббревиатура пяти основных принципов проектирования в объектно-ориентированном программировании
  • Single responsibility (принципы единственной ответственности)
  • Open-closed (открытости / закрытости)
  • Liskov substitution (подстановки Барбары Лиско)
  • Interface segregation (разделения интерфейса)
  • Dependency inversion (инверсии зависимостей)

Solid

Single Responsibility Principle
Принцип единственной ответственности - обозначает, что каждый компонент должен иметь одну обязанность.

sOlid

Open-Closed Principle
Принцип открытости-закрытости - декларирует, что компоненты должны быть открыты для расширения, но закрыты для изменения.

aria-live
Используется для определения важности изменений, которые произошли в элементе.
  • off   (default) — отсутствие приоритета
  • polite   — обозначает низкий уровень приоритета
  • assertive   — указывает на наивысший уровень приоритета
aria-atomic
Необязательный атрибут влияющий на то, в каком объёме вспомогательные технологии объявят об изменениях: это будет весь контент целиком или только его изменившаяся часть.
  • false   (default) — только его изменившаяся часть
  • true   — весь контент целиком
aria-busy
Даёт вспомогательным технологиям знать, обновляется ли сейчас содержимое элемента или нет
  • false   (default) — screen reader не ждет окончания изменений
  • true   — screen reader ожидает окончания изменений
aria-labelledby / aria-describedby
Два схожих атрибута, которые связывают контент (метку) с элементом (картинка, форма и тд) через id.
  • labelledby   — краткое описание элемента
  • describedby   — доп. описание с низким приоритетом
aria-hidden
Отвечает за видимость элемента для вспомогательных технологий.
  • false   (default) — элемент виден
  • true   — элемент не виден
